# Tracking Architecture

Translate the KPI tree into an operating cadence.

The goal is to convert a budget or driver tree into a practical tracking system for management, ownership, and board review.

## Core principle

Not every node belongs in the same review meeting.

Assign each node to the cadence where it becomes decision-useful:
- weekly = early-warning and flow metrics
- monthly = realized outcomes and core operating drivers
- quarterly = structural, strategic, and mix review

## 1. Weekly tracking

Use weekly for metrics that tell management early whether the month or quarter is likely to miss.

Typical weekly metrics:
- pipeline created
- pipeline coverage
- stage conversion snapshots
- opportunities aging
- implementation starts
- implementation backlog
- churn signals / at-risk accounts
- product usage warning signals
- collections delinquency flags
- staffing gaps in revenue-critical functions

Good weekly metrics should be:
- forward-looking,
- actionable,
- operationally owned.

Do not overload the weekly pack with fully lagging financial outputs.

## 2. Monthly tracking

Use monthly for realized outputs and core driver performance.

Typical monthly metrics:
- recognized revenue
- ARR / MRR movement
- new logos
- churn and contraction
- expansion
- pricing realization
- gross margin
- fulfillment or support efficiency
- opex productivity
- collections and cash conversion
- actual vs budget for major driver lines

Monthly review should explain:
- what happened,
- why it happened,
- whether it is structural or timing,
- what actions are required.

## 3. Quarterly tracking

Use quarterly for deeper structural review.

Typical quarterly metrics:
- segment mix shifts
- product mix shifts
- channel productivity
- territory productivity
- cohort quality
- payback trends
- strategic initiative progress
- organizational capacity or structural bottlenecks

Quarterly review should not just repeat monthly outputs. It should assess whether the economic model is changing.

## 4. Required output for every major branch

For each material branch in the tree, specify:
- branch name
- owner
- weekly metrics
- monthly metrics
- quarterly metrics
- threshold for escalation
- likely actions if off plan

## 5. Escalation logic

For each major branch, define:
- what a warning threshold looks like,
- when it becomes a management issue,
- when it becomes a board issue,
- what first action should follow.

Examples:
- pipeline coverage below threshold for 2 consecutive weeks
- implementation backlog above target for 1 month
- gross churn above plan for 2 months
- collections delinquency above threshold with no recovery trend

## 6. Management-review design rules

### Rule A: lead with causes, not outputs
Open with the few driver branches most responsible for performance.

### Rule B: show owner and action
Every material miss should have an owner and a next action.

### Rule C: distinguish structural vs timing
Do not escalate timing noise as structural failure.

### Rule D: keep the pack small
The tracking pack should contain only the KPIs needed to run the business. Avoid metric sprawl.

## 7. Standard tracking pack format

### Weekly pack
- KPI
- current value
- threshold
- trend
- owner
- required action

### Monthly pack
- KPI
- actual
- budget
- variance
- key driver explanation
- owner
- action

### Quarterly pack
- KPI / theme
- current state
- structural implication
- recommendation

## 8. What a good tracking architecture should do

A good output should allow the user to answer:
- what should we monitor every week?
- what explains the month?
- what requires intervention?
- which metrics belong in management review vs board review?
- which branches are truly early-warning indicators?
